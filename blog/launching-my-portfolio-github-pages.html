How I launched this site: GitHub Pages, Cloudflare DNS and a custom .lk domain

Gihan Munasinghe·22 Jul 2026·6 min read

This site — the one you're reading right now — costs me exactly $0/month to run. It deploys itself automatically every time I push a change, serves over HTTPS with a certificate that renews itself, and sits on my own domain, www.gihanmunasinghe.lk. Here's exactly how it's put together, including the two pitfalls that briefly broke the first deployment.

The architecture

There are only three moving parts, and none of them are servers I have to maintain:

  • GitHub Pages hosts the static files on a global CDN, for free.
  • GitHub Actions redeploys the site on every push to main.
  • Cloudflare manages DNS for the domain.

The deployment workflow

The entire CI/CD pipeline is one small workflow file at .github/workflows/deploy.yml: 

name: Deploy to GitHub Pages

on:
  push:
    branches: [main]
  workflow_dispatch:

permissions:
  contents: read
  pages: write
  id-token: write

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/configure-pages@v5
      - uses: actions/upload-pages-artifact@v3
        with:
          path: .
      - uses: actions/deploy-pages@v4

Push to main, and roughly twenty seconds later the change is live worldwide. No build step, no servers, no kubectl.

Pitfall #1: the workflow can't always enable Pages by itself

My first deployment failed with Create Pages site failed: Resource not accessible by integration. The token GitHub gives to workflow runs isn't allowed to switch Pages on for a personal repository. The fix takes ten seconds, but you have to know it exists:

Go to Settings → Pages and set Source to "GitHub Actions" manually, then re-run the workflow. After that one-time step, every deployment is fully automatic.

Pointing a custom domain at it

Two pieces make the custom domain work. First, a file literally named CNAME in the repository root, containing one line — www.gihanmunasinghe.lk — which tells GitHub which hostname to serve. Second, the DNS records in Cloudflare:

  • CNAME record: www → gihanmunasinghe.github.io
  • Four A records on the root (@) pointing to GitHub Pages' IPs: 185.199.108.153 through 185.199.111.153 — so the bare domain works too.

Pitfall #2: Cloudflare's orange cloud

Cloudflare proxies traffic through its own network by default (the "orange cloud"). During initial setup that's a problem: GitHub needs to see your domain directly to verify it and issue the HTTPS certificate. Proxied DNS can leave you stuck in verification limbo — or worse, an infinite redirect loop.

Set the records to "DNS only" (grey cloud) until GitHub shows the domain verified and "Enforce HTTPS" is on. If you want Cloudflare's proxy afterwards, switch its SSL/TLS mode to Full (strict) first.

The result

Total cost: the domain registration. Total maintenance: none. For a portfolio, a product landing page, documentation, or any static site, this stack is very hard to beat — and because the whole thing is just files in a Git repository, the site's own history doubles as its changelog.
