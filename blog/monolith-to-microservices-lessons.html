Monolith → Microservices: 8 lessons I learned the hard way

Gihan Munasinghe·22 Jul 2026·9 min read

I've spent a large part of the last eight years pulling legacy platforms apart — telecom self-care systems serving millions of users, treasury systems where a data error becomes a regulatory conversation, and most recently leading legacy-to-microservices migration at a major bank. Different domains, same gravity: the monolith always fights back.

These are the eight lessons I keep re-learning — the ones I now put in front of every team before we cut the first service.

1. Start with seams, not services

The worst migrations start with an org chart and a whiteboard full of boxes named after teams. The good ones start by finding the seams — places where the domain naturally separates: different rates of change, different data ownership, different scaling profiles.

Before extracting anything, we map modules inside the monolith and enforce boundaries there first. If two "services" would need to call each other for every request, they're one service. You can discover that on a diagram, or you can discover it in production at 2 a.m. — the diagram is cheaper.

2. Strangle it — never big-bang it

Every big-bang rewrite I've seen either quietly died or shipped late missing the features that paid the bills. What works is the strangler fig pattern: put a routing layer in front of the monolith, carve out one capability at a time, and shift traffic gradually.

The under-appreciated benefit: every increment is releasable and reversible. When a newly extracted service misbehaves, you flip the route back and debug in daylight.

3. The database is the real monolith

Splitting the code is the easy 30%. The shared database is where migrations go to die. If ten "microservices" all join against the same tables, you've built a distributed monolith — all of the operational cost, none of the independence.

Data ownership has to move with the service: one service owns a table, everyone else gets the data through its API or its published events. Getting there usually means an ugly interim phase — views, sync jobs, double-writes with reconciliation. Budget for it honestly; it's the longest part of the journey.

4. Contracts first — governance is a feature

Once teams can release independently, the API contract becomes the product. At the bank, one of the highest-leverage things we did was define microservices governance up front: API standards, versioning rules, security requirements, observability expectations, CI/CD gates — adopted across teams before the service count exploded.

Governance sounds bureaucratic until you've debugged an incident spanning five services with five different logging formats, three auth approaches and no correlation IDs. Standards are what make autonomy survivable.

5. Build the runway before you land planes

A monolith needs one pipeline, one dashboard, one deployment. Thirty services need a platform: containerization, CI/CD templates, centralized logging, metrics, tracing, secret management. If that runway doesn't exist by the time you extract your third service, every subsequent extraction gets slower and scarier.

My rule of thumb: platform readiness — Kubernetes/OpenShift, pipeline templates, golden-path service skeleton, observability stack — is a prerequisite for scaling the migration, not a nice-to-have you retrofit later.

6. Embrace events, respect their cost

Synchronous call chains between services recreate the monolith's coupling with network failures added. Kafka (or Pub/Sub) breaks that coupling — billing consumes events at its own pace, and an outage becomes a delay instead of a failure cascade.

But events bring their own tax: duplicates happen, ordering isn't free, and exactly-once is a marketing term. Every consumer we ship is idempotent by design: 

@KafkaListener(topics = "payments.completed")
public void onPaymentCompleted(PaymentEvent event) {
    // Idempotency: the event ID is the guard.
    if (processedEvents.alreadyHandled(event.eventId())) {
        return; // duplicate delivery — safe to ignore
    }
    ledgerService.applyPayment(event);
    processedEvents.markHandled(event.eventId());
}

Boring, mechanical, and it has saved every system I've applied it to.

7. Not everything deserves to be a microservice

Service count is a cost, not a KPI. Each service adds a pipeline, a dashboard, an on-call surface and a version matrix. Some of the best architectural decisions I've made were not extracting something — leaving a stable, rarely-changed capability inside a well-modularized monolith until there was a concrete reason (independent scaling, independent release cadence, separate ownership) to move it.

"Would this be simpler as a module?" is a question that should survive every design review.

8. It's an organizational change wearing a technical costume

The hardest problems weren't in the code. They were ownership arguments, teams afraid to release independently, and stakeholders who expected migration progress and feature delivery from the same headcount. The migrations that stuck were the ones where we treated the operating model — who owns what, who's on call, how releases and reviews work — as part of the architecture.

Conway's law isn't a warning, it's a tool: shape the teams the way you want the system shaped, and the architecture follows with far less friction.

The takeaway

Microservices aren't a destination — they're a trade: coupling exchanged for operational complexity. That trade pays off when you migrate incrementally, move data ownership deliberately, standardize aggressively, and invest in the platform early. Done that way, the monolith doesn't die with a bang; it just quietly runs out of reasons to exist.
